# STEP_2_GACHA_SPEC.md — ガチャ（演出システム）

このアプリの主役。子どもが毎日これのために頑張る中心機能。
抽選ロジックは DATA_MODEL.md の §2.4 で定義済み（レア度テーブル・かけら変換・フォールバック）。
本仕様はその上に乗る「演出（見せ方）」を定義する。

---

## 最重要の設計原則：結果と演出を分離する

- ボタンを押した瞬間に結果（レア度・景品）を確定させる（DATA_MODEL §2.4 の抽選を使う）。
- 確定した結果をもとに、演出シナリオを選んで再生するだけ。
- **演出は結果を変えない。** どれだけ派手に煽っても、内部結果は不変。

→ これにより、演出を後からいくら足してもバグらない／data 追加だけで増やせる。

「煽ったのに普通」「普通かと思ったら激アツ」は、結果に対してどの演出シナリオを選ぶかで作る。

---

## 段階実装

### ステップ2-A：基本の流れ（まずここを完成）

- ボタン→シルエット→マシン登場→カプセル→景品、の一連が動き、景品が正しく付与される状態。
- 1日コンプリート時に1回引ける（`gachaPulled` で当日1回に固定）。
- リカバリー達成でも引ける（`recoveryGrantsGacha` 既定 true。ステップ1.5で導入済み）。
- 月間スペシャルガチャ券があれば、それでも引ける（別枠。§6参照）。
- 抽選 → 景品付与（ポイント/スタンプ/アイテム）、ダブりはかけら変換（既存ロジック）。
- レア度別の最低限の差（カプセル色・光・効果音）。
- スキップ（タップで演出を飛ばして結果へ）。
- 結果画面：景品・レア度表示。ダブりなら「かけら +N」。

### ステップ2-B：期待度演出（ドキドキを乗せる）

2-A の上に、煽り・リーチ・ガセ・大逆転などのシナリオを追加する。

---

## 1. 演出の流れ（シーン構成）

1回のガチャを以下のシーンに分解する。各シーンに「期待度を上げ下げする要素」を差し込める。

| # | シーン | 内容 |
|---|---|---|
| 1 | 引くボタン | 押下（振動・効果音） |
| 2 | シルエット | マシンが影で登場。色がチラ見え（期待度の最初の煽り） |
| 3 | マシン登場 | ガチャガチャマシンが現れ、ハンドルが回る（手応えの音・振動） |
| 4 | 期待度ゾーン | カプセルが出るまでに煽り演出を挿入（2-Bの主戦場） |
| 5 | カプセル落下・オープン | カプセルが開く |
| 6 | 景品お披露目 | 景品ドーン＋レア度に応じたフィニッシュ |

**全体の長さ：** 通常 5〜7秒、大当たり（虹色以上）は 8〜10秒程度。
**モチーフ：** カプセルトイ（ガチャガチャ）マシン。
**スキップ：** いつでもタップで結果へ飛べる（毎日引くので必須）。

---

## 2. 期待度を表現する「演出パーツ」（2-B）

各シーンに差し込める煽り要素。組み合わせでドキドキを作る。

| 演出パーツ | 内容 |
|---|---|
| 色のヒント | シルエット／カプセルの色（白=普通、金=レア寄り、虹=大当たり濃厚） |
| 光のエフェクト | 一瞬キラッ／画面フラッシュ／オーラ |
| リーチ演出 | カプセルがなかなか落ちない・震える・止まりかける |
| 音の段階 | 通常音／予兆音／ファンファーレ前奏 |
| 多段階フラッシュ | 「1回光る→止まる→もう1回光る」のように段階を踏むほど期待大 |
| 背景の変化 | 暗転→虹色など |

---

## 3. 演出シナリオ（結果→どう見せるか）

確定した結果（レア度）ごとに、再生する演出シナリオを選ぶ。
シナリオはデータとして定義し、後から追加できること（このアプリの設計思想）。

### 揺らぎのパターン（2-Bの肝）

| パターン | 内容 |
|---|---|
| 王道 | レア度どおりの派手さ（diamond → 最初から虹・多段フラッシュ） |
| 隠して大逆転 | 高レアなのに途中まで地味 → 最後にドーン（嬉しい裏切り） |
| ガセ（フェイント） | 低レアなのに途中まで虹っぽく煽る → 結果は普通（軽い肩透かし）。出しすぎると萎えるので低頻度 |
| 確定演出 | 特定の演出が出たら「ダイヤ確定」など。ごく稀に出すと特別感が出る |

**重要：** ガセや大逆転も結果は事前確定済み。演出だけを差し替えている。

---

## 4. レア度別フィニッシュ（お披露目の差）

| レア度 | フィニッシュのイメージ |
|---|---|
| ノーマル | シンプルにポン |
| レア | 軽くキラッ |
| スーパーレア | キラキラ＋短い効果音 |
| 激レア | アニメ＋エフェクト強め |
| 虹色 | レインボー演出 |
| 超超超レア | 画面全体の派手な演出 |
| ダイヤモンド | 専用ファンファーレ＋最大級の演出（出た日は特別） |

---

## 5. 演出を後から増やせる構造（必須）

- 演出シナリオ・演出パーツは `src/data/` 配下のデータとして定義。
- 新しい煽り／フィニッシュを足すときは、データに1件追加するだけで使えるようにする。
- 演出再生エンジン（`lib/gachaPlayer.ts` 等）は「シナリオを順に再生する」汎用処理にとどめ、
  個別演出をハードコードしない。
- 2-A では `src/data/gachaVisuals.ts` にレア度別ビジュアル定義を置く。
  2-B でシナリオ定義ファイル（`gachaScenarios.ts`）を追加する。

---

## 6. ガチャを引けるきっかけ（入口の整理）

| きっかけ | 条件 | 消費するフラグ |
|---|---|---|
| デイリー | 1日コンプリートで1回 | `dailyRecords[today].gachaPulled = true` |
| リカバリー | リカバリー達成で1回（`recoveryGrantsGacha` が true の場合） | `dailyRecords[recoveredDate].gachaPulled = true` |
| スペシャルガチャ券 | 月間ごほうびで配布。所持枚数分引ける | `specialGachaTickets -= 1` |

- デイリーとリカバリーは同日重複を防ぐ（`gachaPulled` フラグ）。
- 券は所持数を持ち、引くと1枚消費。通常ガチャとは別枠で引ける。
- 当面、券の抽選テーブルはデイリーと同一（将来「券は高レア率アップ」等にしたくなったら別テーブルを追加できる構造）。

---

## 7. データ構造の追記（DATA_MODEL.md に反映）

```ts
// src/data/gachaVisuals.ts（2-A で追加）
export interface RarityVisual {
  label: string;      // 表示用レア度名（日本語）
  capsuleColor: string;   // カプセル色（CSS色）
  glowColor: string;      // グロー色（CSS色）
  level: number;          // 0〜6（演出強度・音の段数に使用）
}

// src/data/gachaScenarios.ts（2-B で追加）
export interface GachaScenarioDef {
  id: string;
  rarities: Rarity[];     // このシナリオが選ばれうるレア度
  weight: number;         // 重み
  phases: GachaPhase[];   // 演出フェーズの配列
}
```

特記事項：
- 抽選確率・かけら・ポイント量は DATA_MODEL §2.4 の既存値をそのまま使う（変更しない）。
- スペシャルガチャ券で引く場合の抽選テーブルは、当面デイリーと同じ。
- `PullType = "daily" | "ticket"` を `lib/gacha.ts` に定義する。

## 8. 完了条件（AC）

### 2-A

- [ ] コンプリート/リカバリー/券のいずれかで1回引け、当日重複しない。
- [ ] ボタン→シルエット→マシン→カプセル→景品の流れが再生される。
- [ ] レア度別に最低限の見た目差（色・光・音）がある。
- [ ] 景品が正しく付与され、ダブりはかけらに変換される（既存ロジック準拠）。
- [ ] スキップでいつでも結果に飛べる。

### 2-B

- [ ] 結果は押下時に確定し、演出はそれを変えない（結果と演出の分離が守られている）。
- [ ] レア度に応じて演出シナリオが重み付きで選ばれ、毎回同じにならない。
- [ ] 「隠して大逆転」「ガセ」が低頻度で混ざり、期待度の揺らぎが生まれる。
- [ ] ダイヤモンドは専用フィニッシュで特別感がある。
- [ ] 演出シナリオ／パーツがデータ定義で、追加がdata変更だけで済む。
- [ ] 全体の長さが通常5〜7秒・大当たり8〜10秒程度に収まり、スキップ可能。
